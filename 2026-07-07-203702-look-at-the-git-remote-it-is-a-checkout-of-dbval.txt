● Short answer: No — dbval with SlateDB avoids the specific mechanism that made 
  Datahike slow, and a small transaction should cost roughly one object-store 
  PUT instead of 10–15.

  I traced the write path in src/dbval/db.cljc to ground this:

  What dbval actually does per transaction. transact-tx-data (db.cljc:2713)
  accumulates every index tuple — eavt, aevt, avet, teav, so 3–4 keys per datom
  — into a single SlateDB WriteBatch, and commits with one (.write conn batch)
  at the end (db.cljc:2737). Mid-transaction reads never force a round-trip:
  uncommitted keys live in the pending-writes TreeSet that slice merges into
  scans, and scans use DurabilityLevel/MEMORY so they see unflushed data.

  Why the architecture matters. Datahike's cost is structural: a persistent
  B-tree on an object store must path-copy every node from the changed leaf to
  the root, per index, and those PUTs are largely sequential because each parent
  embeds the new child's address — hence 10–15 dependent PUTs and your 1–1.5 s.
  SlateDB is an LSM: a committed batch goes into the in-memory WAL buffer, and
  durability means that buffer being written to the object store as one WAL SST 
  file on the flush interval (default ~100 ms). The default write awaits that
  flush. Crucially:

  - Cost per durable commit is ~½ flush-interval of waiting plus one PUT 
  round-trip, independent of database size, tree depth, and how many indexes
  dbval maintains — all four index ranges are just keys in the same LSM, so they
  ride in the same WAL entry.
  - Concurrent transactions share a WAL flush, so under load latency doesn't
  degrade — multiple commits amortize into one PUT.
  - Memtable→L0 flushes and compaction happen in the background and don't block
  the commit path.

  Expected numbers. Against something like Tigris from Cologne, a durable commit
  should land around 50–200 ms (flush-interval wait + one PUT RTT) rather than
  1–1.5 s. You can push lower by shrinking flush_interval in the DbBuilder
  settings (more, smaller PUTs) or by writing with await_durable = false, which
  returns immediately at the risk of losing the last flush-interval of writes on
  a crash.

  Two caveats. First, the floor is still one object-store PUT per durable commit
  — you get "much better than Datahike," not local-SSD latency; SQLite-on-disk
  will still win on raw commit time. Second, the LSM moves cost to the read
  side: point lookups and range scans can touch multiple SST levels, which is
  where object-store latency could bite instead — SlateDB's block cache and
  bloom filters plus the local db-file cache path you pass to DbBuilder are what
  keep that in check, so read/scan latency is the thing worth benchmarking
  next.
